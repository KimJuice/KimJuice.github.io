上篇文章讲到立项, 下面说说立项之后的那些事…

团队组建

如果所在产品线结构稳定, 已经有相应的执行人员, 那么组建团队这个部分大可忽略; 反之, 则要费一番功夫了. 产品经理并没有行政管理权, 争取资源需要和相关的部门主管/经理申请, 难易程度与公司组织结构和人际关系能力息息相关. 危机情况下, 对于经过会议决定和领导认可的项目可以采取发邮件的方式申请资源, 尽量少用.

计划制定

之前在 PRD 中, 对工作量已经有了一个初评, 现在需要把它细化, 定制出一个可行的实施方案. 常用的估算方法是”三点估算法“: 工作量 = (最乐观 + 最悲观 + 最可能 x 4) / 6. 工作量粒度精确到”1人/天”, 短期项目甚至是”1人/时”. 考虑到工作人员同时有其他任务和集体会议要参加等情况, 计算工作量时应尽量留出富余, 避免连日加班影响工作积极性. 任务相对独立可以采取并行的方式安排, 若任务间相互依赖严重, 也只能更多地串行了.

项目小结

项目发布成功后, 要总结经验, 列出问题和解决方案. 防止忘记细节, 可查看整个项目的周报.

面对变化

  • 变更事件: “需求冻结”后发生的变化, 要慎之又慎.
  • 搭车事件: 一些零散小需求, 不到一周即可完成的, 可适当搭搭顺风车.
  • 紧急事件: 突发紧急状况, 一定要速度站出来共同解决.
  • 其他事件: 例如资源吃紧, 多项目并行的优先级问题等等…

敏捷方法

  • 计划与变化: 在实施过程中不断修正之前不合理的计划.
  • 迭代与任务: 迭代周期内尽量不加任务, 若有时间点内无法完成的任务, 可移出其余部分在下一个迭代周期内完成.
  • 细化需求与测试: 很多需求的细化和补充是在测试中提出的.
  • 发布与交付: 尽早的交付可以看到产品成果, 不断发布是为了步步为营的解决问题, 从核心问题开始, 每次发布都解决一些问题.
